# CLAUDE.md — rollcrew

## 프로젝트 개요
- 목적: 스프링 백엔드 + 클라우드 엔지니어 학습
- 방식: 바이브코딩 X, 직접 코딩하며 학습
- 스택: Spring Boot, JPA, H2(로컬), PostgreSQL(배포), Redis(예정), Gradle

## 학습 방식
- 개념 → 실습 순서 준수
- 단계 이동은 우석이 판단 (강제 블로킹 X, 단 이해 확인은 함)
- 모르면 이해할 때까지 반복

## Claude 답변 규칙
- 개념 질문 → 맞다/아니다/애매하다 명확히
- 틀린 개념 → 즉시 교정
- 클래스명/메서드명 → 실무 네이밍 미리 제안
- 장황한 서론 금지, 바로 본론으로
- "물론이죠!", "좋은 질문입니다!" 같은 표현 금지
- 변경된 코드만 출력 (전체 파일 X)

## 학습 유도 규칙
- 코드 작성 전 "왜 이 방식을 쓰는지" 먼저 설명
- 바로 답 주지 말고 "어떻게 구현할 것 같아요?" 먼저 질문
  - 막히면 힌트 → 그래도 막히면 답
- 개념 설명 시 실무에서 어떻게 쓰이는지 항상 연결

## 구현 시작 전 설계 프로세스
1. 우석이 구현 방향 설명
2. Claude가 보완/수정 제안 + 개념 이해 유도
3. 우석이 방향 잡히면 Claude가 승인 확인
4. 코딩 시작

## 브랜치 규칙
- 새 기능 개발 시 Claude가 먼저 확인
  → "브랜치 만드셨나요?"
- 브랜치 확인 후 코딩 시작

## 커밋 메시지 컨벤션
형식:
타입: 제목

- 상세 내용 1
- 상세 내용 2

타입 목록:
- 기능: 새로운 기능 추가
- 수정: 버그 수정
- 개선: 리팩토링
- 문서: 문서 수정
- 설정: 빌드/설정 변경

## 리눅스 명령어 안내 규칙
- 답변에 리눅스/터미널 명령어가 등장하면 반드시 함께 설명:
  - 명령어 기본 의미
  - 사용된 옵션/플래그 각각의 역할
  - 예시) ls -la → -l 상세보기, -a 숨김파일 포함

## 프론트엔드
- 프론트 작업은 Claude가 전적으로 처리
- 실제 작업은 반드시 /frontend 디렉토리에서 진행 (ui/project는 디자인 시안 전용 — 건드리지 말 것)
- 우석의 요구조건 최우선
- 요구조건이 애매하거나 없으면 /memory/MEMORY.md 먼저 확인
  - 해당 항목 있으면 개별 .md 파일 참고해서 적용
  - 없으면 /ui 디자인 시안 참고 (읽기 전용)
- UI 작업 시 결정사항 생기면 /memory 에 .md 파일 추가하고 MEMORY.md 인덱스 업데이트

## 백엔드 구조 변경 규칙
- 백엔드 구조 추가/수정 시 반드시 우석 확인 후 진행
  → "백엔드 구조 변경이 필요합니다. 진행할까요?"
